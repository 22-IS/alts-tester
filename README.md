# alts-tester

Тестер для лабораторных работ.


## 1. Требования для разработки

- Python 3.13
- Установленный Docker


## 2. Конфигурация

Конфигурация осуществляется через переменные окружения. Для ознакомления с доступными параметрами перейдите в
`src/config.py`.

Важно! Изображение не имеет встроенного Docker сервера. Для корректной работы необходимо дополнительно указать адрес
сервера через переменную окружения `DOCKER_HOST`. По умолчанию обращение проиcходит по адресу 
`tcp://host.docker.internal:2375`. Чтобы этот адрес был доступен на стенде, в Docker Desktop необходимо разрешить 
подключение, поставив галочку в Settings -> General -> Expose daemon on tcp//localhost:2375 without TLS.


## 3. Установка в Jenkins

Если на ноде **отстутствует** Docker сервер, то необходимо установить клиент. Для этого:
1. Необходимо установить плагин Docker Commons (docker-commons).
2. В настройках Jenkins перейти в Tools -> Docker Installations, добавить новую с названием `docker-cli`, выбрать
   `Install automatically` и версию `latest`.
3. В скрипте указать в переменную окружения `DOCKER_HOST` корректный адрес до сервера Docker.
4. Перезапустить Jenkins.

Если на ноде **присутствует** Docker сервер (он чаще всего предоставляется вместе с клиентом), то в скрипте нужно
очистить настройки Docker в секциях `tools` и `environment`.

Далее необходимо создать Сredentials:

| ID                          | Тип                    | Scope  |
|-----------------------------|------------------------|--------|
| `alts-postgres-login`       | Username with password | Global |
| `alts-git-private-repo-url` | Secret text            | Global |

Где:
- `alts-postgres-login` - Имя пользователя и пароль учетной записи СУБД
- `alts-git-private-repo-url` - Ссылка на приватный репозиторий преподавателя (см п.5)

Последними шагами потребуется создать Pipeline, запретить конкурентный запуск, скопировать скрипт, заменив статичные
данные запуска контейнера на корректные, и выполнить первый запуск для загрузки параметров сборки.


## 4. Рекомендации по безопасности

1. Рекомендуется для каждого студента создать отдельный пайплайн из-за невозможности скрытия вывода сборки.
2. Рекомендуется установить `LOG_LEVEL`:
   1. `20` - для студентов (скрыт любой вывод кроме уведомления о фатальной ошибке)
   2. `15` - для преподавателя в тестовом пайплайне (выводятся сообщения о процессе тестирования)
   3. `10` - для разработчиков (дополнительно выводится трассировка и т.д.)
3. Рекомендуется мониторинг Docker контейнеров и процессов на хосте: песочница не гарантирует, что в случае атаки 
   контейнер будет остановлен и удален.


## 5. Создание ссылки на репозиторий

Для работы требуется доступ к контенту репозитория (репозиториев) с правами чтения и записи.

Если в качестве Git провайдера используется GitHub, то ссылка на репозиторий будет иметь вид:
```
https://<токен>@github.com/<пространство>/<репозиторий>.git
```

Чтобы сгенерировать токен, нужно:
1. Перейти в настройки личного профиля GitHub.
2. В левом меню в самом низу найти и перейти в `Developer settings`.
3. В левом меню в `Personal access tokens` перейти в `Fine-grained tokens`.
4. Нажать на кнопку `Generate new token`.
5. Указать имя токена в поле `Token name`.
6. (Опциаонально) Указать описание токена в поле `Description`.
7. Выбрать время истечения токена в поле `Expiration` (по умолчанию токен активен в течение 30 дней после создания).
8. В `Repository access` выбрать `Only select repositories`.
9. В `Select repositories` выбрать нужные репозитории.
10. В `Add permission` найти и выбрать `Contents`.
11. После добавления в `Permissions` напротив `Contents` поменять `Read-only` на `Read and write`.
12. Нажать на кнопку `Generate token` и подтвердить действие.

Таким образом, был сгенерирован токен с помощью которого можно читать и изменять содержимое выбранных в
`Select repositories` репозиториев через API GitHub. Токен будет автоматически деактивирован через срок, выбранный в 
поле `Expiration`.

Например, `username` нашего аккаунта - `example`, нужный репозиторий - `repo`, тогда полная ссылка на репозиторий примет
вид:
```
https://github_pat_XXXX@github.com/example/repo.git
```
Где:
- `github_pat_XXXX` - сгенерированный токен
- `example` - пространство имен, т.е. `username` аккаунта
- `repo` - название репозитория

Важно! В контексте примера токен был сгенерирован для репозитория `example/repo`, т.е. при генерации этого токена, в
списке `Select repositories` этот репозиторий был выбран.

Другие Git провайдеры, в том числе и self-hosted, также поддерживают создание токенов для управления репозиториями,
однако информацию по генерации **их** токенов нужно искать в **их** документации.


## 6. Создание тестов

Для тестирования лабораторной работы, после идентификации студента, в классе App автоматически вызывается функция
`_test_labX_varY()`, где:
- `X` - номер л.р.
- `Y` - номер варианта

Функция должна вернуть одно целое число от 0 до 100 - количество баллов за проверку.

Пример функции для тестирования первого варианта л.р. 1:
```python
def _test_lab1_var1() -> int:
    logging.debug('_test_lab1_var1()')
    return 100
```

Общие рекомендации для написания тестов:
1. Функции тестирования должны располагаться в порядке возрастания в секции `TESTS`.
2. После объявления функции должен быть произведен `logging.debug()` с именем функции.
3. После каждого уменьшения количества баллов должен быть вызван `self._log_internal()` со `stage` = `test`
   и причиной уменьшения баллов. Другими словами, причина, почему студент не набрал 100 баллов, всегда должна
   быть обоснована в `self._log_internal('test', '<причина>')`.

В тестах может потребоваться контент репозитория студента. Для этого можно воспользоваться объектом
TemporaryDirectory `self._public_repo_dir`. Другие доступные объекты можно найти в функции `_perform()` 
в секции `BASIC`.

В приложении также имеется возможность запуска песочницы для тестирования bash-скриптов в Docker контейнере
через `self._run_script()`, где:
1. `content` - содержимое скрипта
2. `args` - входные аргументы скрипта

Результат выполнения - `tuple[int, str, str]`, где:
- `tuple[0]` - целое число, код выполнения
- `tuple[1]` - строка, stdout
- `tuple[2]` - строка, stderr

Для проверки корректности выполнения скрипта достаточно:
1. Чтобы код выполнения был равен 0.
2. Чтобы stdout соответствовал ожидаемому выводу.

Важно! stderr вопреки названию может содержать логи, а не ошибки, поэтому полагаться на его проверку нельзя.
